Your First Look: A Travel Agent

Let's peek at how one of these apps actually works. Here's the core of the AI Travel Agent from the Streamlit app:

CODE 
researcher = Agent(
    name="Researcher",
    role="Searches for travel destinations...",
    model=OpenAIChat(id="gpt-4o"),
    tools=[SerpApiTools()]
)

planner = Agent(
    name="Planner",
    role="Generates travel itinerary...",
    model=OpenAIChat(id="gpt-4o")
)
PLAIN ENGLISH

Create a "Researcher" agent whose job is to search the web for travel information...

Give it access to GPT-4o (the AI brain) and SerpAPI (a tool to search Google)...


Create a "Planner" agent that takes the research and organizes it into an itinerary...

It also uses GPT-4o but doesn't need search tools — it just processes the info.

Notice how the code defines two specialized agents, each with a specific job. This is a fundamental pattern you'll see throughout the repository.

Starter AI Agents: Your First Steps

Learn how simple agents work by building practical tools

What Makes an Agent "Starter"?

A starter agent is like a single-purpose tool. Think of a hammer — it does one thing really well. These agents are:

  • Single-purpose (one clear job)
  • Self-contained (all code in one file)
  • Easy to modify for your own needs

Perfect examples: a travel planner, music generator, or data analyst. Each does exactly what it says on the tin.

Anatomy of a Starter Agent

Every agent has three essential parts, like a recipe with three ingredients:

CODE 
agent = Agent(
    name="Data Analyst",
    role="Analyzes CSV data...",
    model=OpenAIChat(id="gpt-4o"),
    tools=[
        PandasTools(),
        PlotlyTools()
    ],
    instructions=[
        "Load the CSV file",
        "Analyze patterns",
        "Create visualizations"
    ]
)
PLAIN ENGLISH

The agent needs a name and description of its job...

Pick which AI model to use (GPT-4o is powerful and popular)...

Give it tools it can use — like Pandas for spreadsheet work and Plotly for making charts...

Provide step-by-step instructions so it knows what to do with the data.

💡
Key Insight

This pattern — defining who the agent is, what brain it uses, and what tools it has — is universal across all agent frameworks. Once you understand this structure, you can build with any framework.

The Agent-Tool Relationship

Think of an agent like a chef, and tools like kitchen equipment. The chef (agent) knows how to cook, but needs knives, pans, and mixers (tools) to actually make the meal.

Common tools in starter agents:

  • SerpAPI: Search Google (like having a research assistant)
  • PandasTools: Work with spreadsheets and data (like Excel, but in code)
  • FileTools: Read and write files (like having a filing cabinet)
  • WeatherAPI: Get current weather data (self-explanatory!)

Advanced Agents & Multi-Agent Teams

When one agent isn't enough — building teams that collaborate

Why Teams Over Single Agents?

Imagine trying to build a house with just one person who has to be the architect, plumber, electrician, and carpenter. They'd be overwhelmed and make mistakes.

The same principle applies to AI agents. Complex tasks need specialized teams where each agent focuses on what they do best.

💡
Key Insight

This mirrors real software engineering: breaking complex problems into smaller, focused pieces. It's called "separation of concerns" — one of the most important ideas in building reliable systems.

Team Patterns

Multi-agent teams typically follow one of these organizational patterns:

→

Sequential

Agents work in order, like an assembly line. Agent A finishes, passes to Agent B, who passes to Agent C.

⚡

Hierarchical

One "manager" agent delegates tasks to specialist agents, like a project manager assigning work.

🔄

Loop

Agents take turns refining the output, like editors reviewing and improving a document.

Real Example: VC Due Diligence Team

The AI VC Due Diligence Agent Team is a perfect example. It mimics how real venture capitalists evaluate startups:

CODE 
market_analyst = Agent(
    role="Market Research Analyst",
    # Researches market size, competitors
)

financial_analyst = Agent(
    role="Financial Analyst",
    # Evaluates business model, revenue
)

tech_analyst = Agent(
    role="Tech Analyst",
    # Assesses technical feasibility
)

synthesizer = Agent(
    role="Investment Synthesizer",
    # Combines all analyses into recommendation
)
PLAIN ENGLISH

Create a specialist who only looks at market conditions and competition...

Another expert focuses purely on money: revenue models, burn rate, path to profitability...

A tech expert evaluates if the product is technically sound and can scale...

Finally, a senior partner takes all three reports and makes an investment recommendation.

Each agent has deep expertise in their area. The final agent synthesizes their findings — just like a real investment committee.

RAG: Teaching Agents About Your Data

Build AI that knows about your specific documents and information

The Problem RAG Solves

ChatGPT knows about Shakespeare and Python programming. But it knows nothing about your company's sales data, your personal notes, or that PDF report from last week.

RAG fixes this. It's like giving the AI a library card to your personal collection of documents.

📚
The Library Metaphor

RAG works like a library with a smart catalog system. When you ask a question, the system finds the relevant books (your documents), reads the important pages, and uses that information to answer you.

How RAG Works in 4 Steps

Here's the complete journey of a question through a RAG system:

CODE 
# Step 1: Convert documents to embeddings
embeddings = embed_documents(pdf_files)

# Step 2: Store in vector database
vector_db.add(embeddings)

# Step 3: When user asks a question
question = "What was Q3 revenue?"
relevant_docs = vector_db.search(question)

# Step 4: Send docs + question to LLM
answer = llm.ask(
    context=relevant_docs,
    question=question
)
PLAIN ENGLISH

First, convert your documents into a special numerical format that captures their meaning...

Store these in a database designed for finding similar content (called a "vector database")...

When someone asks a question, search the database for the most relevant document sections...

Send both the relevant documents AND the question to the AI, so it can answer using your specific information.

Types of RAG in the Repository

The repository includes different RAG flavors for different needs:

⛓️

Basic RAG Chain

Simple question → search docs → answer. Great for starting out and understanding the fundamentals.

🔄

Agentic RAG

The AI decides when to search docs vs. using its own knowledge. Smarter but more complex.

🔍

Corrective RAG

Checks if retrieved docs are actually relevant, searches web if needed. Most sophisticated approach.

MCP Agents & Voice AI

Connecting agents to real-world tools and making them talk

What is MCP?

MCP is like a universal adapter for AI agents. Before MCP, every tool needed custom code to connect. With MCP, tools expose a standard interface that any agent can use.

Think of it like USB ports — before USB, every device had different connectors. Now everything uses the same standard.

🔌
Why MCP Matters

MCP is to AI agents what APIs are to web services — a standard way to connect different systems. This is how you build production-grade agents that integrate with your existing tools.

MCP Agents in Action

The repository includes agents that connect to real services:

🐙

GitHub MCP Agent

Read repos, create issues, review PRs — your AI coding assistant that actually understands your codebase.

📑

Notion MCP Agent

Query your Notion workspace, create pages, update databases — like ChatGPT but for your company wiki.

🌐

Browser MCP Agent

Control a real browser, click buttons, fill forms — web automation powered by AI understanding.

Voice AI: Agents That Speak

Voice agents add a new dimension — they can listen and respond with speech. Perfect for hands-free workflows or accessibility.

The voice pipeline has three key components:

  • Speech-to-Text: Convert your voice to text
  • LLM Processing: The AI thinks and decides what to say
  • Text-to-Speech: Convert the response back to voice

Building Your Own LLM App

Putting it all together — from idea to working application

The Decision Framework

You've seen starter agents, multi-agent teams, RAG systems, MCP integrations, and voice. How do you choose what to use for YOUR project?

Here's a decision tree to guide you:

🤔

Ask: Single task or complex workflow?

Single task: Start with a starter agent. Complex: Consider multi-agent team.

📚

Ask: Need custom data?

Yes: Add RAG. No: The LLM's built-in knowledge is enough.

🔌

Ask: Need to control external tools?

Yes: Use MCP integrations. No: Standard tools are fine.

Real-World Architecture Example

Let's design an agent for a real scenario: A customer support assistant for an e-commerce company.

What it needs to do:

  • Answer questions about products (from catalog)
  • Check order status (from order system)
  • Handle returns (update database)
  • Escalate complex issues to humans

Architecture decision:

ARCHITECTURE 
# Multi-agent team
product_expert = Agent(
    tools=[RAG(catalog_docs)]
)

order_specialist = Agent(
    tools=[MCPTool("order-api")]
)

returns_handler = Agent(
    tools=[MCPTool("returns-db")]
)

coordinator = Agent(
    role="Routes to specialists"
)
REASONING

Why multi-agent: Three distinct knowledge domains — products, orders, returns.

Why RAG for products: Catalog changes frequently, needs custom data.

Why MCP for orders/returns: Need to read/write to existing company systems.

Why coordinator: Routes customer to the right specialist, just like a receptionist.
